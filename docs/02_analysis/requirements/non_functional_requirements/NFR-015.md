# NFR-015: Checkpoint State by Epoch and Cap Per-Node Storage at Design Time

## Statement
The Ledger system shall create epoch checkpoints and shall define a per-node storage cap at design time.

## Rationale
[[BPA_Report#4.2 Consolidated NFR Registry|BPA NFR-STORE-01]] defines the storage target as epoch checkpointing with a per-node cap defined at design time. [[nasir_2022_batch03]] identifies storage scalability as a critical blockchain NFR, while [[al_bassam_2018]] and [[sonnino_2021]] both describe signed checkpoint chains and audit-friendly state histories. This makes the storage policy directly traceable and reviewable without adding a gap.

## Acceptance Criteria
- [ ] The system produces checkpoints at epoch boundaries or an explicitly documented equivalent cadence.
- [ ] The design-time storage cap for each node is documented before deployment.
- [ ] The checkpointing policy supports audit and recovery use cases without relying on ad hoc storage growth.

## Notes
This requirement covers storage policy and checkpoint cadence, not retention scheduling or backup tooling.
