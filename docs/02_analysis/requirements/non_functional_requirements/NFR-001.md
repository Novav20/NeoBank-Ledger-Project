# NFR-001: Maintain BFT Availability with Rapid View Change

## Statement
The Ledger system shall maintain availability under Byzantine fault conditions by operating a BFT consensus cluster with $3F+1$ nodes and completing leader view change within 30 seconds.

## Rationale
[[BPA_Report#4.2 Consolidated NFR Registry|BPA NFR-AVAIL-01]] defines the availability target as $3F+1$ nodes, BFT, and view-change <30s. [[barger_2021]] supports the $2F+1$ quorum out of $3F+1$ nodes, the view-change sub-protocol, and node catch-up behavior; [[berger_2023b]] supports view change and faulty-leader replacement; [[chuen_2017]] supports $n \ge 3f + 1$ BFT availability. The recovery-window detail remains a BPA synthesis point, so it is tracked in the gap register.

## Acceptance Criteria
- [ ] The system continues to make progress when up to $F$ nodes are faulty in a $3F+1$-node cluster.
- [ ] When the active leader becomes unavailable, the system completes a view change and resumes progress within 30 seconds.
- [ ] Nodes that fall behind can synchronize missing ledger data without manual intervention.

## Notes
The exact <30s recovery window is a BPA-derived threshold. The source summaries reviewed for this extraction confirm the quorum and view-change behavior but do not restate the bound explicitly. That provenance issue is recorded in the gap register.
