# FR-005: Periodic Audit Checkpointing

## Statement
The Ledger system MUST generate and publish a signed cryptographic checkpoint (Merkle root) of the Event Log at defined intervals (epochs) to facilitate continuous oversight.

## Rationale
This requirement is grounded in [[BPA_Report#4.1 Foundational Principles (Core Business Rules)|BPA Section 4.1]] and the BPA auditability posture that requires evidence-grade logging and non-repudiation, and it supports UC-03 by making the checkpoint evidence available for audit verification.

## Acceptance Criteria
- [ ] The system publishes a signed checkpoint at the configured epoch interval.
- [ ] The published checkpoint represents the Event Log as a cryptographic Merkle root.
- [ ] The checkpoint is available for continuous oversight and later audit verification.

## Notes
The delivery pattern for audit evidence is hybrid: periodic push of checkpoints plus pull-based transaction proofs on demand.
