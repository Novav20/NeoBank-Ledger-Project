# Copilot Response: ADR-002 & ADR-003 Generation (Finality & Sharding)

## Executive Summary
I created ADR-002 and ADR-003 to lock the core architecture baseline. PBFT is selected for deterministic finality on the core consensus cluster, and MSSP is selected for sharding with a 20%-25% m-node ratio to reduce cross-shard coordination cost.

## Analysis / Findings
The finality decision needs to satisfy the BPA requirement for deterministic settlement and fork-freedom. PBFT is the clearest fit because it gives immediate finality instead of probabilistic settlement, which aligns with B2B trust expectations. RAFT is unsuitable for the core trust path because it only handles crash faults.

The sharding decision needs to preserve throughput without forcing every cross-shard interaction through heavyweight coordination. MSSP is the right fit because m-nodes can participate in multiple consensus zones, which reduces the pressure to rely on 2PC for every boundary crossing. The selected 20%-25% ratio keeps the topology bounded while leaving room for scale.

## Proposed Plan / Solution
1. Use PBFT as the default consensus model for the core cluster.
2. Keep the PBFT cluster bounded to control quadratic messaging overhead.
3. Use MSSP with a 20%-25% m-node ratio for sharded execution and storage.
4. Move next into HLD preparation with the ADR-001/002/003 stack as the architecture input.
